## 诊断结论

我查到这次不是手机号或验证码的问题：`18657433310 / 123456` 的服务端验证请求已经成功返回了用户和超级管理员身份。

真正失败点在后续跳转：登录成功后进入 `/app`，页面没有读到当前会话，于是立刻又跳回 `/auth?redirect=/auth`。这通常发生在预览 iframe / 跨站环境里，`SameSite=Lax` Cookie 不能稳定作为登录态。继续只修 Cookie 会反复踩坑。

## 目标

1. 彻底修复当前登录后又回登录页的问题。
2. 接入腾讯云真实短信验证码。
3. 保留明确错误提示：发送失败、验证码错误、过期、登录态异常时都给用户可操作提示。

## 实施计划

### 1. 先把登录态改成稳定方案

当前登录态主要依赖 Cookie。我会改成“双通道会话”：

- 服务端仍写 httpOnly Cookie，适合正式独立访问域名。
- 同时创建一个随机会话 token，只返回给前端一次。
- 前端把 token 存在本地，用全局 server function middleware 自动附加到请求头。
- 服务端收到请求后优先读 Cookie；Cookie 不可用时，用请求头 token 到数据库校验并解析用户。

这样即使预览环境阻止 Cookie，`/app` 和项目接口也能识别登录态，不再跳回 `/auth`。

### 2. 新增安全会话表

新增一张仅服务端可访问的会话表，用来存储：

- token_hash
- user_id
- expires_at
- created_at
- last_used_at

前端永远拿不到数据库里的 hash，只持有一次性生成的随机 token。退出登录时清理 Cookie、本地 token 和服务端会话。

### 3. 接入腾讯云短信发送

新增腾讯云短信服务端调用：

- `sendSmsCode` 生成 6 位随机验证码。
- 将验证码 hash 存入数据库，设置 5 分钟过期。
- 调用腾讯云 SMS API 发送短信。
- `verifySmsCode` 校验最新未过期验证码，限制错误尝试次数，成功后标记已使用并登录。

不会把腾讯云密钥写进代码，会通过安全 Secret 表单配置。

### 4. 需要你在腾讯云准备这些信息

你需要在腾讯云控制台准备：

1. 开通「短信 SMS」服务。
2. 创建或确认一个短信应用，拿到 `SDKAppID`。
3. 申请短信签名，拿到「短信签名内容」。
4. 申请短信模板，例如内容类似：
   - `您的验证码为{1}，{2}分钟内有效，请勿泄露。`
5. 创建访问密钥，拿到：
   - SecretId
   - SecretKey

后续我会用安全 Secret 输入框收集这些值，不要直接发在聊天里。

### 5. 需要配置的 Secrets

我会请求你填写这些安全变量：

- `TENCENTCLOUD_SECRET_ID`
- `TENCENTCLOUD_SECRET_KEY`
- `TENCENT_SMS_SDK_APP_ID`
- `TENCENT_SMS_SIGN_NAME`
- `TENCENT_SMS_TEMPLATE_ID`
- `TENCENT_SMS_REGION`，默认可用 `ap-guangzhou`

如果你的模板需要两个参数，我会传：验证码、有效分钟数。

### 6. 登录页提示和兜底

登录页会改成真实短信文案：

- 发送成功：提示验证码已发送。
- 发送失败：显示腾讯云返回的可读原因。
- 验证码错误：提示重新输入。
- 验证码过期：提示重新获取。
- 会话异常：提供「重新登录」按钮清理状态。

我也会移除“开发期固定 123456”的误导提示。是否保留 `18657433310 / 123456` 作为仅预览环境超级管理员兜底，可以在实现时作为临时安全开关；正式发布建议关闭。

### 7. 验证方式

完成后我会验证：

1. 当前模拟超级管理员能成功进入 `/app`，不再回跳。
2. 登录后项目列表接口能识别用户。
3. 退出登录后本地 token、Cookie、服务端会话都被清理。
4. 腾讯云 Secrets 配置后，真实短信发送链路可用。

## 腾讯云操作指引

在腾讯云侧你按这个顺序做：

1. 进入腾讯云短信 SMS。
2. 创建短信应用，记录 `SDKAppID`。
3. 添加短信签名，审核通过后记录签名名称。
4. 添加正文模板，建议模板包含验证码参数，比如 `{1}`，如需有效期则加 `{2}`。
5. 进入访问管理 CAM，创建一个用于此应用的密钥。
6. 等我实现后，在安全 Secret 表单里填写上面的 5-6 个值。
